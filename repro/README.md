# Reproducing the paper

> The LaTeX source is on the **`paper`** branch, not main. Paths below that begin with `paper/`
> resolve only there (`git switch paper`); the regeneration commands themselves run on either
> branch, since they read from `1/`–`4/`.

Every table in `paper/main.tex` is regenerated by one of the commands below. All experiments are
seeded; re-running reproduces the numbers bit-for-bit, except the wall-clock columns in
`tab:weff`, `tab:breakeven` and `tab:csp-solved`, which are hardware- and load-dependent — the
*ratios* reproduce, the absolute milliseconds do not. `tab:weff`'s timings are medians over 5
repetitions; the run-to-run spread is 6–21%.

**Sections below are keyed to LaTeX `\label` names, not table numbers.** Labels survive inserting a
table; numbers do not, and nothing in the build checks a companion file. To resolve a label to its
current number:

```bash
grep -oE '\\newlabel\{tab:[a-z-]+\}\{\{[0-9]+\}' paper/main.aux \
  | sed 's/\\newlabel{//; s/}{{/ -> Table /; s/}//' | sort
```

Each assignment is an independent [uv](https://github.com/astral-sh/uv) project. Sync before use.

One input is not generated: §4 runs on a live OpenStreetMap extract, and OSM is edited
continuously, so the snapshot the paper used is committed at `1/data/dhaka_graph.graphml.gz`
(Overpass base timestamp `2026-04-21T12:54:14Z`). `load_or_download()` reads it without unpacking.
Nothing downloads unless you pass `force_download=True`, which would give you a *newer* graph and
different node counts — see `1/data/README.md`.

## `tab:weff`, `tab:breakeven` — informed search (§4)

```bash
cd 1 && uv sync
uv run python scripts/instrumented_batch.py     # ~4 min
uv run python ../repro/search_analyze.py
```

`instrumented_batch.py` fixes three measurement problems in the stock batch driver
(`dhaka_pathfind.analysis.batch`):

1. **Setup folded into search.** `build_heuristic_aux` runs two full O(|E|) passes before an
   informed search expands anything. The driver's outer clock includes it; the library's own
   `SearchResult.metrics.runtime_ms` excludes it (its timer starts after the aux build). We record
   both, and time the aux build separately.
2. **The timed regions were not the same region.** `astar()` computes `_suffix_costs()` — an
   O(path_len²) re-evaluation of the multi-factor edge cost — plus a per-path-node heuristic
   evaluation *inside* its timed region, for the `heuristic_mean_abs_gap` diagnostic
   (`algorithms.py:288-303`). Worth a median of 26 ms, all of it charged to one arm. We time it on
   the identical path and subtract. Only `ucs()` and `weighted_astar()` are free of
   post-processing: `bidirectional_ucs()` and `greedy_best_first()` each call `_path_cost()`
   inside their timed regions too (`algorithms.py:244` and `:419`), because neither carries a
   g-value to the target. That call is O(path_len) and costs well under a millisecond — see
   `path_cost_probe.py` below — so we measure it but leave it in.
3. **Single-shot timings.** Search times vary 6–21% run to run. We take 5 repetitions per
   (pair, configuration) and report medians, with the aux build memoised so repetitions measure
   search rather than setup.

It also writes `greedy_invariance.csv`, the check behind Proposition 1.

`search_analyze.py` prints both tables: the weight sweep (`tab:weff`) and the paired break-even
analysis (`tab:breakeven`). The latter pairs over the 10 landmark pairs and reports a 95% CI on
each per-query saving, because a break-even is a large constant divided by a difference of two
noisy timings — one of its four comparisons is not resolvable at n=10, and the script says so
rather than printing a point estimate.

Outputs land in `1/outputs/results/`: `instrumented_42.csv`, `setup_cost.csv`,
`greedy_invariance.csv`, `astar_postproc.csv`.

The two `_path_cost()` figures quoted in §4.2 come from a separate probe, which re-runs only the
two affected arms on the same ten pairs so that the headline timings are not disturbed:

```bash
cd 1 && uv run python ../repro/path_cost_probe.py     # ~1 min
```

## `tab:csp-solved` — constraint satisfaction (§5)

```bash
cd 2 && uv sync
.venv/bin/python ../repro/csp_rerun.py          # ~35 min
.venv/bin/python ../repro/csp_analyze.py
```

Note two things about this assignment's own artifacts, which is why the paper re-runs rather than
reading `2/results/`:

- `2/results/REPORT.md` tabulates two solvers, `bt_cbj_fc_mrv` (conflict-directed backjumping) and
  `bt_ac3_mrv` (AC-3 preprocessing), that this codebase cannot produce. There is no CBJ
  implementation anywhere in the tree — only a `backjumps` field referenced by
  `SolverResult.pretty()`, which itself raises `AttributeError` because `SolverStats` has no such
  field. `fuel_csp/algorithms/ac3.py` does define an `AC3Solver`, but it is absent from
  `ALL_SOLVERS` and fails to import (`fuel_csp.constraints.conflict_set` does not exist).
- `2/results/experiments_{raw,summary}.csv` were produced by a different run than `REPORT.md` —
  the node counts disagree (e.g. `basic_backtracking` at n=10: 54,431 in the CSV, 24,611 in the
  report).

`csp_rerun.py` runs only the five solvers that are registered and executable, over 10 instance
sizes × 8 seeds at a 5 s budget, and records a `censored` flag per run so the paper can separate
budget-limited cells from algorithm-limited ones.

## `tab:pso-ablation`, `tab:pso-gen` — population-based search (§6)

```bash
cd 3 && uv sync
./run.sh all       # 23 tests + experiments, ~5 min
```

Outputs: `3/results/tables/A{1..5}*.csv`. `tab:pso-ablation` is `A2_communication_ablation.csv`
plus `A2_communication_tests.csv`; `tab:pso-gen` is `A4_generalisation.csv`; the topology and
sensitivity numbers in §6.2 are `A3_topology*.csv` and `A5_sensitivity.csv`.
`A1_convergence_summary.csv` and `A1_convergence_tests.csv` back the GA-beats-PSO comparison in
§6.2 (W = 106, p = 0.008) — and note that §6.1's margin of PSO *against random search* (9.05 min,
p = 1.9e−9) is in `A1_convergence_tests.csv` too, not in the A2 pair the ablation table is keyed
to.

## `tab:rl-baselines`, `tab:rl-ce`, `tab:sweep` — reinforcement learning (§7, Appendix B)

```bash
cd 4 && uv sync
./run.sh all       # 28 tests + experiments, ~10 min
```

Outputs: `4/results/tables/B{1..6}*.csv`. `tab:rl-baselines` is `B2_baselines.csv`; `tab:rl-ce` is
`B3_learning_summary.csv` (with the per-seed values, including the seed on which certainty
equivalence loses, in `B3_learning_raw.csv`); the misspecification crossing point in §7.3 is
`B4_model_misspecification.csv`; `tab:sweep` is `B5_hyperparameters.csv`.

Note that `B5` runs at 1,500 episodes × 3 seeds while `B3` runs at 4,000 episodes × 5 seeds, so
the two tables' absolute regrets are not comparable. The sweep selects the tuned configuration;
the tuned configuration is then re-run at the full budget for `tab:rl-ce`.

## Tables with no script

Two tables are hand-authored summaries rather than experimental output, and nothing regenerates
them:

- `tab:testbed` (§3) — the four problems, the mechanism each ablation tests, and the budget held
  fixed. Prose.
- `tab:metrics` (§8) — each family's customary metric, what it fails to charge, and the measured
  consequence. Every number in it is carried over from the sections above.

## Checking this file

`repro/check_labels.py` verifies that the labels cited here all exist in `paper/`, that every table
label in `paper/` is mentioned here, and that this file contains no bare "Table N" references. Run
it after any edit that adds or moves a table:

```bash
python repro/check_labels.py
```

## Building the paper

```bash
git switch paper
cd paper
pdflatex main && bibtex main && pdflatex main && pdflatex main
```

No `siunitx` dependency; the source compiles on a base TeX Live install.
