# eponym-ablation

**Is the mechanism a classical AI method family is named for the mechanism that does the work?**

Four problems in one setting — urban mobility in Dhaka — each instantiating a different method
family, each implemented from scratch, each ablated at a matched compute budget. Every table in the
paper is regenerated by [`repro/`](repro/); the LaTeX source lives on the
[`paper`](../../tree/paper/paper) branch, to keep main to the code and the experiments.

| Folder | Problem | Family | Eponym under test | Matched budget |
|--------|---------|--------|-------------------|----------------|
| [`1/`](1/) | Multi-factor routing over the Dhaka OSM graph | Informed search | the heuristic | per-query wall clock |
| [`2/`](2/) | Urban fuel-crisis allocation | Constraint satisfaction | ordering + propagation | 5 s per instance |
| [`3/`](3/) | Campus shuttle timetabling | Population-based search | the population | 3,030 objective evaluations |
| [`4/`](4/) | Campus-gate signal control | Reinforcement learning | the learning | 400,000 transitions |

## What the ablations found

- **Search.** The three "different" heuristics are positive scalar multiples of one function, so
  greedy best-first is provably invariant to the choice (identical paths on 10/10 queries) and A\*
  over them is a one-parameter weight sweep (ρ = −1.0000 against nodes and search time).
  Nodes-expanded ranks A\* above uniform-cost search by 2.15×; wall clock ranks it below by 6.18×,
  because 91.6% of A\*'s runtime is a one-off O(|E|) heuristic calibration. The fastest exact
  method here is uninformed.
- **Constraints.** Ordering and propagation mostly prune as advertised (−3 to −5 nodes, p ≤ 1e−4 —
  but MRV on its own prunes nothing, p = 0.27) and run 12–14× slower for it; they change which
  instances are solvable in 1 case out of 80, and that one adversely. Past the point where search
  stops completing, the node metric inverts and measures throughput.
- **Population.** Thirty non-communicating particles are indistinguishable from random search
  (p = 0.22); one particle is *worse* than random search. The social term alone is worth 7.6
  minutes of every student's expected wait. The population is not the mechanism — the channel is.
- **Learning.** On identical data — same seeds, same 400,000 transitions, same order — planning on
  an estimated model beats a *tuned* Q-learner (1.74% vs 2.72% regret; 89.9% vs 65.7%
  action-optimality). Planning with arrival rates wrong by 10× still wins. Only a model that
  believes no traffic exists loses.

The cross-family reading: the mechanisms that survive budget-matched ablation are those that
convert a per-unit cost into an amortised one, and each subfield's customary metric is the one
that declines to charge for amortisation.

## Setup

Each problem is a self-contained [uv](https://github.com/astral-sh/uv) project with its own
`pyproject.toml` and lockfile:

```bash
cd 1 && uv sync    # or: cd 2 && uv sync, cd 3 && uv sync, cd 4 && uv sync
```

Each folder has a `run.sh` entry point and its own `README.md`. Every experiment takes an explicit
seed and reproduces bit-for-bit; there is no unseeded randomness in any solver.

See [`repro/README.md`](repro/README.md) to regenerate the paper's tables. To rebuild the PDF,
switch to the `paper` branch (`git switch paper`) and run
`pdflatex main && bibtex main && pdflatex main && pdflatex main` in `paper/`.

## Provenance

The four problems originate as independent coursework assignments for the AI Lab at the Department
of Computer Science and Engineering, University of Dhaka, each specified before its results were
known and each carrying a written instruction to report disconfirming results rather than retune
the instance. Several of the headline findings are disconfirmations of what the assignment led us
to expect, and they are stated first rather than buried.

## A note on large files

Problem 1 runs on an OpenStreetMap extract of Dhaka. OSM is edited continuously, so the snapshot
the paper's §4 numbers come from is committed — `1/data/dhaka_graph.graphml.gz`, 6.6 MB, Overpass
base timestamp `2026-04-21T12:54:14Z` — and is read without unpacking. A clone reproduces those
node counts offline; nothing downloads unless you ask it to. The uncompressed graph (~58 MB) and
the derived Parquet edge table are generated from it and are **not** tracked; see `1/data/README.md`.
